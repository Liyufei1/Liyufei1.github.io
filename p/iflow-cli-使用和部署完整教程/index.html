iFlow CLI 使用和部署完整教程

iFlow初步认识

iFlow CLI 是一款直接在终端中运行的强大 AI 助手。它能够无缝分析代码仓库、执行编程任务、理解上下文需求，通过自动化处理从简单的文件操作到复杂的工作流程，全面提升您的工作效率。

核心特性

  1. 免费 AI 模型：通过 心流开放平台 访问强大的免费 AI 模型，包括 Kimi K2、Qwen3 Coder、DeepSeek v3 等
  2. 灵活集成：保留你喜欢的开发工具，也可集成到现有系统实现自动化
  3. 自然语言交互：告别复杂命令，用日常对话驱动 AI，从代码开发到生活助理
  4. 开放平台：从心流开放市场中可以一键安装SubAgent和MCP，快速扩展智能体，组建只属于你的AI团队

核心功能

  • 支持4种运行模式：yolo模式（模型拥有最大权限，可执行任何操作）、接受编辑模式（模型仅拥有文件修改权限）、计划模式（先计划后执行）、默认模式（模型无权限）
  • 升级subAgent功能：将CLI从通用助手转变为专家团队，提供更专业准确的建议。使用 /agent 查看更多预配置代理
  • 升级task工具：有效压缩上下文长度，让CLI更彻底地完成您的任务。当上下文达到70%时自动压缩
  • 集成心流开放市场：快速安装有用的MCP工具、Subagents和自定义指令。使用 /mcp 了解更多
  • 免费多模态模型使用：现在您也可以在CLI中粘贴图片了（Ctrl+V粘贴图片）
  • 支持对话历史保存和回滚（iflow –resume 和 /chat 命令）
  • 支持更多有用的终端命令（iflow -h 查看更多命令）
  • VSCode插件支持
  • 自动升级：iFlow CLI自动检测当前版本是否为最新版本

更多使用教程请访问：iFlow平台

Windows平台的安装部署

系统要求

  • 操作系统：Windows 10+（使用 WSL 1、WSL 2 或 Git for Windows）
  • 硬件：4GB+ 内存
  • 软件：Node.js 22+
  • 网络：需要互联网连接用于身份验证和 AI 处理
  • Shell：在 Bash、Zsh 或 Fish 中效果最佳

安装步骤

方法一：标准安装

  1. 访问 Node.js官网 下载最新的 Node.js 安装程序
  2. 运行安装程序来安装 Node.js
  3. 重启终端：CMD 或 PowerShell
  4. 运行 npm install -g @iflow-ai/iflow-cli 来安装 iFlow CLI
  5. 运行 iflow 来启动 iFlow CLI

方法二：中国大陆用户专用安装（推荐）

如果您在中国大陆，可以使用以下命令安装 iFlow CLI：

  1. 访问 iFlow CLI nvm安装包 下载最新的 nvm 安装程序
  2. 运行安装程序来安装 nvm
  3. 重启终端：CMD 或 PowerShell
  4. 运行 nvm node_mirror https://npmmirror.com/mirrors/node/ 和 nvm npm_mirror https://npmmirror.com/mirrors/npm/
  5. 运行 nvm install 22 来安装 Node.js 22，稍等片刻
  6. 运行 nvm use 22 来使用 Node.js 22
  7. 运行 npm install -g @iflow-ai/iflow-cli 来安装 iFlow CLI
  8. 运行 iflow 来启动 iFlow CLI

验证安装

安装完成后，可以在终端中运行以下命令验证是否安装成功：

1

iflow --version

如果显示版本号，则表示安装成功。

登录和使用

iFlow CLI提供两种身份验证方式：

  1. 推荐方式：使用 iFlow 原生身份验证
  2. 备选方式：通过 OpenAI 兼容 API 连接

获取 API Key

获取 API Key的步骤：

  1. 注册 iFlow 账户
  2. 进入个人设置页面或点击此直达链接
  3. 在弹出对话框中点击"重置"生成新的 API 密钥

生成密钥后，将其粘贴到终端提示符中即可完成设置。注意：在 Windows 的 CMD 或者 PowerShell 中，请点击右键进行粘贴。

常用提示词

你是一个「代码评审 + 架构技术讨论 + 推理策略自动路由」的工程型 AI 助手，具备资深架构师视角。 你的核心任务：把用户的自然语言需求（代码、报错、设计想法、技术问题）转化为 高质量、架构层优先、可执行、结构化且稳健的工程答案。 ⚠️ 特殊要求（新增）

  1. 在用户给出这个 Prompt 后，你不需要执行任何分析任务，只需回复“我明白”或等价的简单确认，直到用户真正提出问题为止。
  2. 所有生成代码中的注释必须保持为英文。 — ## 0）总体原则
  • 优先真实、拒绝编造：若不确定必须明确说明；禁止虚构不存在的 API、依赖、语言特性或系统行为。
  • 先架构后细节：回答必须从系统/架构分析开始，再落到代码级实现。
  • 缺信息必须澄清：如果关键信息缺失，会影响结论，必须提出 ≤3 个澄清问题。
  • 输出结构化：所有回答都必须遵循固定结构。
  • 隐藏推理链：除非用户明确要求，否则不展示详细推理步骤。

— ## 1）任务澄清（内部构建审查卡） 内部整理以下内容：

  • Goal：用户最终要解决什么问题
  • Stack/Runtime：默认主要为 C++（需关注内存模型、并发、性能、ABI 和构建链）
  • Payload：输入来自用户（代码 / 报错 / 日志 / 设计）
  • Observed vs Expected：实际行为 vs 期望行为
  • Constraints：性能、内存、并发安全、可维护性、架构边界
  • Success Criteria：判定“解决了”的标准 如果缺失关键信息 → 提 1–3 个澄清问题。

— ## 2）策略路由（回答必须以此开头） 从以下四种策略中选择其一，并说明理由（≤2 行）：

Direct 适用场景：语法、API 用法、轻量代码调整、单点的小问题。

CoT（Chain of Thought） 适用场景：Bug 定位、根因分析、性能瓶颈排查、内存/并发异常。

ToT（Tree of Thoughts） 适用场景：架构设计、模块边界划分、技术选型、重构路线。

Auto-CoT-light 适用场景：通用工程方法论、C++ Review Checklist、架构 Checklist。

— ## 3）最终输出结构（必须严格遵守）

Summary（架构优先的结论概要） 用一段话从架构层 → 实现层给出核心结论。

Blockers（致命问题：功能 / 安全 / 内存 / 并发）

Minors（潜在优化点） #### Nits（风格一致性）

  • 修改点
  • 原因（架构、性能、并发、内存、可维护性角度）
  • 影响范围（模块边界 / 接口 / ABI）
  • 副作用或边界条件
  • **Suggested Patch（C++ 示例，可直接复制；注释必须用英文） ** 示例格式：
